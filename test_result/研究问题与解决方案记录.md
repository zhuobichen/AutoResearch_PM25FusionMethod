## 问题1：VNA实现错误

### 问题描述
Agent代码中使用了`VNAFusionCore`类，但调用方式错误，导致VNA的R²计算结果极低（0.2而非0.8+）。

### 错误代码
```python
# 错误：VNAFusionCore需要特定列名格式
vna_core = VNAFusionCore(k=30, power=-2)
vna_core.fit(train_df)  # 期望mod, bias, r_n等列
vna_pred = vna_core.predict(test_df[['Lon', 'Lat']].values)
```

### 原因
`VNAFusionCore.fit()`期望的DataFrame列名格式为：
- `mod`: CMAQ模型值
- `bias`: 偏差 (O - M)
- `r_n`: 比值 (O / M)
- `Conc`或`obs`: 观测值

但实际传入的是标准列名`Conc`, `CMAQ`, `fold`等，导致KeyError。

### 正确实现
```python
# 正确：直接使用NNA类
from Code.VNAeVNAaVNA.nna_methods import NNA

nn = NNA(method='nearest', k=15, power=-2)
nn.fit(train[['Lon', 'Lat']].values, train['Conc'].values)
vna_pred = nn.predict(test[['Lon', 'Lat']].values)
```

### 关键教训
1. 不要假设第三方类的接口格式与你的数据匹配
2. 优先使用底层类（NNA）而非高层封装（VNAFusionCore）
3. 测试时先验证基准方法是否正确

## 问题2：GPR计算过慢

### 问题描述
使用GPR进行10折交叉验证时，每个fold需要拟合GPR，导致整体运行时间超过数小时。

### 原因
- GPR的时间复杂度为O(n³)，n~1300时很慢
- 10折 × 15天 = 150次GPR拟合

### 解决方案
1. **减少n_restarts_optimizer**：从2改为0（不优化核参数）
   ```python
   gpr = GaussianProcessRegressor(kernel=kernel, n_restarts_optimizer=0, alpha=0.1)
   ```

2. **固定核参数**：不进行优化
   ```python
   kernel = RBF(length_scale=1.0) + WhiteKernel(noise_level=1.0)
   gpr = GaussianProcessRegressor(kernel=kernel, alpha=0.1)  # 不调用optimizer
   ```

3. **减少测试天数**：先跑5天而非15天

### 关键教训
1. 研究阶段用简化配置快速迭代
2. 最终验证时再使用完整配置

## 问题3：十折验证数量不匹配

### 问题描述
有时十折交叉验证的总预测数不等于原始数据量。

### 原因
```python
# 错误：如果某个fold的test为空，可能漏掉
for fold_id in range(1, 11):
    train = day_df[day_df['fold'] != fold_id]
    test = day_df[day_df['fold'] == fold_id]
    if len(test) == 0:
        continue  # 这里跳过了，但结果字典是空的
```

### 正确实现
```python
# 正确：确保每个fold都被处理
results = {}
for fold_id in range(1, 11):
    train = day_df[day_df['fold'] != fold_id]
    test = day_df[day_df['fold'] == fold_id]
    
    if len(test) > 0:
        # 进行预测
        results[fold_id] = {'y': y_test, 'pred': pred}

# 汇总时只取有结果的fold
all_y = np.concatenate([results[f]['y'] for f in range(1, 11) if f in results])
```

### 验证方法
```python
total_pred = sum(len(results[f]['y']) for f in range(1, 11))
assert total_pred == len(day_df), f"预测数量不匹配: {total_pred} vs {len(day_df)}"
```

### 关键教训
1. 每次验证后检查总数是否匹配
2. 用assert或print进行基本 sanity check

## 问题4：日期索引计算错误

### 问题描述
从日期字符串获取CMAQ数据索引时出错。

### 错误代码
```python
# 错误：enumerate返回的是DataFrame的行号，不是日期索引
day_df['CMAQ'] = [get_cmaq_at_site(...) for i, (_, r) in enumerate(day_df.iterrows())]
cmaq_data[i]  # i是DataFrame行索引，不是日期索引
```

### 正确实现
```python
# 正确：从日期字符串计算索引
day_str = '2020-01-15'
day_idx = int(day_str.split('-')[2]) - 1  # 0-indexed

day_df = monitor_df[monitor_df['Date'] == day_str].copy()
day_df['CMAQ'] = [get_cmaq_at_site(r['Lon'], r['Lat'], lon_cmaq, lat_cmaq, cmaq_data[day_idx])
                  for _, r in day_df.iterrows()]
```

### 关键教训
1. 数据处理时明确索引来源
2. 不要混用行索引和逻辑索引

## 问题5：多日数据处理时的fold泄露

### 问题描述
使用多日数据联合训练时，可能出现训练集和测试集站点重叠的问题。

### 原因
```python
# 潜在问题：多日数据中同一站点出现在多天
# 如果训练集包含站点A的day1，测试集包含站点A的day2
# 可能造成信息泄露
```

### 注意事项
1. 确保站点级别的分离，而非样本级别的分离
2. 当前使用fold_split_table按站点划分，是正确的
3. 多日数据时，确保同一天的所有站点 fold 划分一致

### 关键教训
1. 时空数据要注意空间和时间上的信息泄露
2. 验证时要同时检查空间分离和时间分离

## 问题6：Agent实现不可靠

### 问题描述
Agent生成的方法实现可能有隐藏的错误。

### 示例
ST-CRK Agent报告的VNA R²=0.2024，但实际应为0.8861。

### 教训
1. Agent生成代码后，必须手动验证关键部分
2. 用已知的基准方法（VNA）验证实现是否正确
3. 不要完全信任Agent的输出，要做抽查验证

### 验证流程
```python
# 1. 先用简单数据验证实现
# 2. 对比已知正确实现（直接用NNA vs 你的VNA）
# 3. 确认基准方法正确后再进行对比测试
```

## 总结：最佳实践

### 代码实现
1. **优先使用底层类**：NNA比VNAFusionCore更可靠
2. **验证基准方法**：每次测试先确认VNA/eVNA/aVNA结果合理
3. **数据索引清晰**：明确区分行索引、日期索引、fold索引
4. **sanity check**：用assert和print检查数量匹配

### 研究流程
1. **简化配置快速迭代**：先用少量数据测试
2. **完整配置最终验证**：确认简化版本有效后，再跑完整
3. **手动抽查**：不100%信任Agent输出
4. **记录问题**：遇到问题及时记录，避免重复

### 报告结果
1. **注明实现方式**：让结果可复现
2. **标注数据来源**：哪天、哪种数据
3. **保留原始代码**：方便复查
