```
    │              │              │
    │              │              ├─ 通过 → Stage 3 (12月整月)
    │              │              │              │
    │              │              │              └─ 全部通过 → 创新成立

    │              └─ 未通过 → 创新不足，打回重设
    │
    └─ 未通过 → 打回 [Phase 3] 重新设计
```

# 十二、版本管理与终止条件

## 12.1 版本管理

| 对象 | 命名格式 | 保存规则 |
|------|----------|----------|
| 方法方案 | ideas_V1.json, V2... | 永久保存 |
| 代码 | workspace/innovation/V1/, V2/... | 永久保存 |
| 测试结果 | results/innovation/V1/, V2/... | 永久保存 |
| 错误日志 | error/xxx_YYYYMMDD.log | 永久保存 |

**【禁止】覆盖任何历史版本**

## 12.2 自动终止条件

| 条件 | 说明 |
|------|------|
| 创新成立 | R²提升≥0.01 且 RMSE≤最优基准 |
| 人类明确停止 | 人类说"停止" |

## 12.3 人类干预入口

### 方式一：修改配置文件（立即生效）

```json
// test_result/流程配置.json
{
  "max_iterations": 10,
  "min_r2_improvement": 0.01,
  "vna_r2_min": 0.70,
  "vna_r2_max": 0.95,
  "stagnation_threshold": 3,
  "test_days": ["2020-01-01", "2020-01-02"],
  "enabled_methods": ["VNA", "eVNA", "aVNA", "RK-Poly"]
}
```

### 方式二：强制采用特定方案

```json
// test_result/最终方案标记.json
{
  "final": true,
  "method": "RK-Poly",
  "round": 15,
  "note": "人工确认为最优方案"
}
```

### 方式三：跳过基准校验

```
创建文件：error/skip_baseline_check_YYYYMMDD.flag
（文件存在即表示跳过基准带校验）
```

### 方式四：手动指定基准方法

```json
// test_result/基准方法配置.json
{
  "primary_baseline": "eVNA",
  "secondary_baselines": ["VNA", "aVNA"]
}
```

## 12.4 【禁止】自行终止

```
即使连续多轮无提升、超过10轮仍未成功，也【禁止】自行终止。
```

# 附录A：创新方法排除规则

> **重要**：参见项目根目录 `PM2.5_CMAQ融合方法创新排除.md`

## A.1 需要排除的方法类型

**加权集成方法（Stacking/Ensemble with learned weights）**：

| 应排除的方法 |
|-------------|
| SuperStackingEnsemble | EnhancedStackingEnsemble | UltimateStackingEnsemble |
| FeatureStackingEnsemble | MultiLevelStackingEnsemble | LogRatioEnsemble |
| V6-Ensemble-V1 | OptimizedTripleEnsemble | TripleEnsemble |

**排除原因**：
| 原因 | 说明 |
|------|------|
| 迁移性能不强 | 权重在特定日期学习，不同日期可能需要不同权重 |
| 效果提升有限 | 往往比最优子方法提升很小(<0.005)，甚至不如 |
| 可解释性差 | 负权重含义不明确，物理意义不清楚 |
| 复杂但无用 | 实现复杂，参数众多，难以部署 |

## A.2 应该保留的方法类型

| 特征 | 描述 |
|------|------|
| 单模型 | 单一方法解决问题 |
| 物理可解释 | 数学/物理意义清晰 |
| 显著提升 | R²提升 ≥ 0.02 |
| 可迁移 | 权重不随日期剧烈变化 |

**鼓励的创新方向**：
| 类型 | 示例 | 物理意义 |
|------|------|---------|
| 新偏差校正 | RK-Poly | CMAQ偏差非线性 |
| 新空间建模 | GPR降尺度 | 残差空间相关性 |
| 新融合策略 | 协同克里金 | 多变量空间协同 |

## A.3 创新判定流程（补充）

```
1. 新方法是否使用权重学习（Ridge/Lasso等）？
   → 是 → 排除
   → 否 → 继续

2. 新方法是否有物理可解释性？
   → 否 → 排除
   → 是 → 保留（创新成立）
```

## A.4 典型案例

| 方法 | R² | 结论 |
|------|-----|------|
| SuperStackingEnsemble | 0.8571 | **应排除**（加权集成，RK-Poly效果更好） |
| RK-Poly | ~0.85-0.88 | **应保留**（物理意义清晰，单模型） |

# 附录D：衍生方法创新评估原则（2026-04-17 新增）

## D.1 核心观点

**当某个基础创新方法已通过验证后，衍生方法的测试价值必须从论文贡献度衡量。不必要的方法应剔除，避免论文冗余。**

> 适用场景：当项目存在一个已验证的核心创新方法（如 PolyRK），需要评估其衍生方法是否值得进一步测试时，适用本原则。

## D.2 衍生方法分类与评估

### 同族方法（原理相同，参数/核函数不同）

| 评估项 | 判定标准 | 结果 |
|--------|----------|------|
| 是否有明确创新点 | 核函数改进有物理意义（如RBF→Matern） | ✅ 保留 |
| | 参数调整无明确物理意义 | ⚠️ 需验证后再定 |
| | 实质无效创新（如样条回退多项式） | ❌ 排除 |

### 不同族方法（原理不同）

| 评估项 | 判定标准 | 结果 |
|--------|----------|------|
| 是否是完全不同的技术路线 | IDW vs GPR，图网络 vs 传统统计 | ⚠️ 可探索 |
| 探索价值 | 效果是否明确优于基础方法 | ✅ 仅当明确优于时保留 |

### 评估示例

| 衍生方法 | 基础方法 | 核心区别 | 评估 | 结论 |
|----------|----------|----------|------|------|
| AdvancedRK | PolyRK | RBF核→Matern核 | GPR核函数改进，有物理意义 | ✅ 保留 |
| RobustRK | PolyRK | OLS→Huber | 抗异常值提升 | ✅ 保留 |
| MSRK | PolyRK | 单尺度→多尺度 | 复杂度增加 | ⚠️ 可选 |
| PSK | PolyRK | 多项式→样条 | 样条回退，无效创新 | ❌ 排除 |
| CSPRK | PolyRK | 全局→分层 | 分层边界不合理 | ❌ 排除 |
| PG-STGAT | PolyRK | GPR→图注意力 | 不同技术路线 | ✅ 保留（已验证） |

## D.3 论文贡献度评估标准

```
┌────────────────────────────────────────────────────────────────┐
│              衍生方法是否值得测试？                              │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 是否有明确创新点？                                          │
│     └─ 原理改进 vs 参数调整 vs 完全同质                         │
│                                                                 │
│  2. 创新点是否已被同类方法覆盖？                                 │
│     └─ AdvancedRK (Matern) vs PolyRK (RBF) → 有差异，可保留    │
│                                                                 │
│  3. 测试复杂度 vs 论文贡献度                                     │
│     └─ 4阶段验证耗时数小时，方法堆砌无意义                      │
│                                                                 │
│  4. 论文是否需要？                                              │
│     └─ 核心创新(PolyRK) + 基准对比(VNA) 已足够                  │
│     └─ 衍生方法仅当有明确优于PolyRK时才列出                      │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

## D.4 评估决策树

```
新方法评估
    │
    ├─ 是否与已有方法同质？（核函数/参数不同但原理相同）
    │   ├─ 是 → 是否有明确改进？
    │   │   ├─ 是（有物理意义）→ ✅ 保留
    │   │   └─ 否（复杂度增加但无明确优势）→ ⚠️ 需验证后再定
    │   └─ 否 → 是否是完全不同的技术路线？
    │       ├─ 是 → ⚠️ 可探索（论文贡献视情况）
    │       └─ 否 → ❌ 排除
    │
    └─ 论文是否已有核心创新？
        ├─ 是 → 衍生方法必须有明确优于核心方法时才列出
        └─ 否 → 评估新方法是否可作为核心创新
```

## D.5 通用精简原则

**当存在已验证的核心创新方法时**：

| 方法类别 | 处理方式 |
|----------|----------|
| 核心创新方法 | 保留，作为论文核心贡献 |
| 原理改进变体 | 有明确物理意义时保留（如核函数改进） |
| 参数调整变体 | 效果明确优于核心方法时才保留 |
| 完全不同路线 | 作为消融实验对比，可选 |

**论文结构建议**：
```
方法对比：
- 基准方法：VNA / eVNA / aVNA
- 核心创新：XXX (原理说明)
- 变体对比：YYY (与核心方法的区别)
- 备选探索：ZZZ (不同技术路线)
```

> **注**：具体方法示例见 D.2 评估示例表格

## B.4 基准方法多阶段验证要求

### 基准方法验证代码模板

```python
def validate_baseline_methods_multistage():
    """
    基准方法多阶段验证

    输出：test_result/基准方法/benchmark_multistage.json
    """
    methods = ['VNA', 'eVNA', 'aVNA', 'Downscaler']
    stages = {
        'pre_exp': ('2020-01-01', '2020-01-05'),
        'stage1':  ('2020-01-01', '2020-01-31'),
        'stage2':  ('2020-07-01', '2020-07-31'),
        'stage3':  ('2020-12-01', '2020-12-31'),
    }

    results = {}
    for method in methods:
        results[method] = {}
        for stage_name, (start, end) in stages.items():
            all_y_true, all_y_pred = [], []
            current_date = start
            while current_date <= end:
                # 每天十折验证
                daily_true, daily_pred = ten_fold_for_day(method, current_date)
                all_y_true.extend(daily_true)
                all_y_pred.extend(daily_pred)
                current_date += timedelta(days=1)

            # 合并计算整体指标
            metrics = compute_metrics(all_y_true, all_y_pred)
            results[method][stage_name] = metrics

    # 保存结果
    with open('benchmark_multistage.json', 'w') as f:
        json.dump(results, f, indent=2)

    return results
```

# 附录C：参数迁移性评估

## C.1 核心矛盾

| 方案 | 问题 |
|------|------|
| 只跑5天预实验调参 | 参数只对这几天有效，其他天用同样参数结果变差 |
| 从第一天就跑全部天数 | 太贵，失去了预实验的意义 |
| 物理意义赋值 | 很多参数（尤其是偏差校正系数）根本没有物理意义可以查表 |

## C.2 两阶段参数策略

```
第一阶段：物理初始化
↓
用物理意义给参数赋"合理初值"
（比如 Kriging 的 range 初值，可以用 PM2.5 典型空间相关距离）
↓
第二阶段：有限数据验证
↓
在 1-2 天上做参数微调
→ 验证"这套初值 + 少量数据 → 能泛化到其他天"
↓
跨天测试（看参数是否迁移）
```

## C.3 四元数组的隐性第5维度

| 维度 | 评判标准 |
|------|---------|
| 1.定量提升 | R² ≥ 历史最佳 + 0.01 |
| 2.物理可解释性 | 参数能用物理意义初始化，或在有限数据下稳定收敛 |
| 3.跨日期一致性 | 在≥3个不同天测试，参数不剧烈震荡 |

**第2/5条重点**：不是"参数完全来自物理"，而是"**参数有物理锚点，不怕小数据扰动**"

## C.4 能排除的方法类型

| 类型 | 特征 | 论文风险 |
|------|------|----------|
| 数据饥渴型 | 依赖大量数据才能拟合，数据少了就崩 | 审稿人挑战 |
| 日期波动型 | 参数在好日期和坏日期之间剧烈波动 | 说明在拟合日期特征 |

## C.5 参数迁移性验证

| 验证场景 | 做法 |
|----------|------|
| 单日参数 vs 多日平均 | 对比同一方法在单日学习和多日联合学习下的表现 |
| 固定参数方法 | 如 T1=35, T2=75 基于国标，天然具有迁移性 |
| 建议策略 | 优先设计具有物理意义的固定参数，减少数据依赖 |

**关键洞见**：参数对某一天的数据特别敏感（小扰动导致大变化）→ 本质上是在"拟合那一天"而非"建模"
